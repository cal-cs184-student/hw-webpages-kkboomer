Part 1: Ray Generation and Scene Intersection (20 Points)

Walk through the ray generation and primitive intersection parts of the rendering pipeline.

We start by looping through every pixel ($x$, $y$ coordinates), shooting multiple random rays through a single pixel to reduce aliasing. First, we convert the screen space of the pixel into world space. We normalize the screen coordinates so they sit between 0 and 1, then map them onto a virtual “sensor plane” in 3D camera space. Next, we create a ray originating from the camera’s position, pointing towards that specific point on the sensor plane. To translate this into the 3D scene, we multiply the ray's direction vector by a camera-to-world rotation matrix (c2w) to put it in world space. We also assign the ray a min_t and max_t to ensure no intersections happen out of bounds. We shoot this ray into the scene. If it hits a primitive, we update the max_t of the ray to equal the distance of the hit. By constantly shrinking max_t with every intersection, we ensure that we only render the closest object, naturally handling depth and occlusion.


Explain the triangle intersection algorithm you implemented in your own words.

Now that we can shoot a ray, we need to check if it intersects with an object. For triangles, we use the Möller-Trumbore algorithm. Recall that a ray can be defined as $R(t)=O+tD$, where $O$ is the origin and $D$ is the direction vector. A point inside a triangle can be defined using its 3 vertices ($P_0, P_1, P_2$) and barycentric coordinates:

$$P(u,v)=(1-u-v)P_0 + uP_1 + vP_2$$

To check for an intersection, we set the ray equation equal to the triangle point equation. Using Cramer’s rule, we rearrange the variables to solve for $t$, $u$, and $v$ simultaneously. Once we have those values, we perform two strict checks:

  1. Is the hit within valid bounds? The distance $t$ must be between our clipping planes (ray.min_t <= t <= ray.max_t).
  2. Is the hit inside the triangle? The barycentric coordinates must be valid ($u \ge 0$, $v \ge 0$, and $u+v \le 1$).

If both pass, the ray successfully hit the triangle! We then use those exact $u$ and $v$ values to interpolate the surface normal at the hit point by blending the normals of the 3 vertices: $N=(1-u-v)N_0 + uN_1 + vN_2$. Finally, we populate our Intersection struct with this interpolated normal $N$, the material's BSDF, and the distance $t$.

Part 2: Bounding Volume Hierarchy (20 Points)

Walk through your BVH construction algorithm. Explain the heuristic you chose for picking the splitting point.

My BVH construction algorithm (BVHAccel::construct_bvh) is a top-down, recursive function. When called, it follows these steps:

  1. Compute the Bounding Box: We first compute the bounding box. Iterating through all primitives that the current call has access to, we expand a bounding box big enough to encapsulate all of them. We also compute the bounding box of their centroids, which is used later for splitting nodes.
  2. Create the Node: We then create a new BVH Node and initialize it with this expanded bounding box.
  3. The Base Case (Leaf Nodes): Next, we check if the number of primitives in the current call is less than or equal to max_leaf_size. If it is, the new node becomes a leaf node. We assign primitive iterators to the new node’s start and end pointers, and simply return the node.
  4. The Recursive Step: If that check does not pass, we must split the node. We determine the splitting axis and the split point.
  5. Partitioning: Using std::partition, we reorganize the primitives into two groups depending on whether their centroid is to the right or left of the split point.
  6. Edge Case Handling: If a partition is empty, we force a split exactly down the middle of the array (start + (count / 2)) to ensure the tree keeps breaking down geometry and maintains its structure.
  7. Recursion: Finally, we recursively call the function on the left and right partitions, assigning the returned nodes to the left and right pointers of the new node, and return it.

For the splitting heuristic, I decided to divide the primitives along the longest axis of the overall bounding box’s extent, using the average of the primitive centroids along that axis as the exact split point.

First, we compare the $x$, $y$, and $z$ extents of the bounding box to determine the longest geometric axis, which guarantees we split along the widest physical dimension, keeping the bounding boxes as cube-like as possible.

To find the split point, we calculate the average position for all primitive centroids along the chosen axis. This helps distribute the primitives more evenly, creating a much more balanced tree that is significantly faster to traverse while rendering.

Part 3: Direct Illumination (20 Points)

Walk through both implementations of the direct lighting function.

We have 2 main ways of implementing the directing lighting.

1. Uniform Hemisphere Sampling

We first create a local coordinate system where the Z-axis and the surface normal are aligned. We then perform Monte Carlo integration by looping through a set number of samples. For every sample, we call hemisphereSampler->get_sample() to generate a random incoming direction vector ($w_i$) uniformly distributed across the hemisphere above the hit point.

We convert $w_i$ to world space by multiplying it by the object-to-world matrix, and create a new shadow ray originating from the hit point. We offset the shadow ray's origin slightly by setting min_t = EPS_F to prevent the ray from intersecting the surface it just bounced off of (preventing shadow acne). The shadow ray is then shot into the BVH. If an intersection happens, we query the emission of the hit object. If the object is a light source, it will return a non-zero radiance ($L_i$).

Finally, we calculate the contribution of this sample using the reflection equation: $L_i \times f(w_{out}, w_{in}) \times w_i.z \times 2\pi$. (Note: We multiply by $2\pi$ because dividing by the uniform hemisphere PDF of $1/(2\pi)$ is mathematically equivalent). We end by averaging all the samples together to get the final direct lighting color.

2. Importance Sampling

Instead of sampling randomly, we now only shoot rays directly at the actual light sources. We loop through all the light sources in the scene. For point or directional lights, we only take one sample, while for area lights, we take ns_area_light samples.

For every sample, we call the light’s sample_L function, which gives us the incoming radiance ($L_i$), the light direction ($w_i$), the distance to the light, and the PDF for the sample. After converting $w_i$ to object space, we check to see if the light is actually in front of the surface ($w_i.z > 0$); if it is not, we ignore it.

We then construct a shadow ray pointing at the light, making sure to set max_t = distToLight - EPS_F. This ensures that the ray stops exactly at the light source and is not falsely occluded by an object behind it. I used the bvh->has_intersection() function to test for occlusion, since we only care if the light is blocked, not what blocked it. If the path is clear, we calculate the contribution using the rendering equation and accumulate it into a running total of outgoing radiance.

Compare the results between uniform hemisphere sampling and lighting sampling in a one-paragraph analysis.

Uniform hemisphere sampling produces a very noisy image. This is due to the fact that rays are fired randomly into the hemisphere, with a large majority of them missing the light entirely and returning zero. Because of this, the renderer must rely on a massive amount of samples to accurately guess the light's true brightness. Furthermore, point lights are nearly impossible to render with this method, as the statistical chance of a randomly fired ray hitting an infinitesimally small point is practically zero. On the other hand, importance sampling actively targets the light sources, guaranteeing that every single un-occluded ray returns valid light data. This allows the sampling to converge almost immediately, producing smooth gradients and crisp soft shadows without wasting computational power on rays that hit dark, empty space.


Part 4: Global Illumination (20 Points)

Walk through your implementation of the indirect lighting function.

The core of global illumination is handled by the at_least_one_bounce_radiance() function, which allows rays to bounce multiple times to accumulate indirect light. Because light can bounce infinitely, this is implemented as a recursive function with strict termination conditions.

  1. The Base Case: We begin by calculating the direct illumination at the current intersection using one_bounce_radiance() and adding it to our total outgoing radiance ($L_{out}$). If the current ray’s depth is 1, this means the ray is on its final permitted bounce, so we immediately return $L_{out}$ and stop recursing.
  2. Russian Roulette (Termination): To efficiently prevent infinite recursion without artificially darkening the image, we use Russian Roulette. We set a continuation probability of 0.7 (meaning a 70% chance that a ray survives the bounce) and flip a weighted coin. If the coin returns false, we terminate early. Note: Because the project specifies that we must trace at least one indirect bounce (if max_ray_depth > 1), we only subject the ray to this coin flip on the second bounce and beyond.
  3. Sampling the Bounce: If the ray survives, we call the material's sample_f() method. This returns a random incoming direction vector in object space ($w_i$), the probability of that direction being chosen ($pdf$), and the material’s reflectance ($f$).
  4. Shooting the Recursive Ray: After converting $w_i$ to world space by multiplying it by the object-to-world matrix (o2w), we create a new bounce_ray originating from the current hit point. We offset it slightly (min_t = EPS_F) to prevent shadow acne, and—crucially—decrement the ray’s depth by 1.
  5. The Recursive Equation: We then shoot this new ray into the BVH. If it hits another object, we recursively call at_least_one_bounce_radiance() on the new intersection to gather the incoming light ($L_{in}$) bouncing from the rest of the scene. Finally, we apply the rendering equation: we multiply the incoming light by the reflectance and the cosine of the incoming angle, and divide by both the sampling PDF and the Russian Roulette continuation probability:
$$L_{out} \mathrel{+}= \frac{L_{in} \times f \times w_i.z}{pdf \times cpdf}$$

We add this evaluated indirect light to $L_{out}$ and return the final accumulated color.

Part 5: Adaptive Sampling (20 Points)

Explain adaptive sampling. Walk through your implementation of the adaptive sampling.

Adaptive Sampling

With Monte Carlo based path tracing, some pixels converge to their true color much faster than others. Objects like flat walls may only need a small amount of samples to converge, while soft shadows may need all given samples to converge. Adaptive sampling is an optimization that tracks the color of the pixels as it renders; when the color stops f luctuating, we assume convergence and stop shooting rays.

Here is how I implemented it in raytrace_pixel:

Instead of storing an array of every single ray color, we maintain a running sum of the ray’s illuminance and squared illuminance ($s_1$ and $s_2$ respectively), and a counter for the number of rays shot so far ($n$). Additionally, I now only perform the convergence check every samplesPerBatch number of rays to save on computation.

When we do check, we calculate the mean ($\mu$) and standard deviation ($\sigma$) of the samples gathered so far:

$$ \mu = \frac{s_1}{n} $$ $$ \sigma = \sqrt{\frac{1}{n-1} \left( s_2 - \frac{s_1^2}{n} \right)} $$

Using those values, we calculate the confidence interval variable $I$:

$$ I = 1.96 \times \frac{\sigma}{\sqrt{n}} $$

If $I \le \text{maxTolerance} \times \mu$, we are within the 95% interval of confidence that the pixel has converged, at which point we break out of the loop. Once we reach the end of the loop by either breaking early or taking ns_aa amount of samples, we divide the total accumulated radiance by the real number of samples ($n$) rather than the max number of samples, so as to not decrease overall luminance. We end by recording $n$ in the sampleCountBuffer so we can accurately generate the heat map.
